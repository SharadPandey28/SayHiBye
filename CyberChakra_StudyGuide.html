SLIDE 04 Android Security Architecture — 6 Layers ▼

WHAT THIS SLIDE SHOWS

A layered pyramid of Android's 5 main layers + SELinux cutting across all + three extra protections at the bottom.

Layer 1 — Application Sandbox
Every app gets a unique Linux UID. App A literally cannot read App B's memory — the kernel enforces this. Apps declare permissions in their manifest; users grant them at runtime. Your Facebook app cannot access your banking app's data.
Layer 2 — Android Framework (Java API)
High-level services: ActivityManager (what's running), PackageManager (app installs), WindowManager (screen). All app-to-service calls go through Binder IPC. If your app doesn't have a permission, the framework rejects the call before it even reaches the service.
Layer 3 — ART Runtime + Native Libraries
ART compiles and runs app bytecode. Native libraries (libstagefright for media, OpenSSL for crypto, SQLite) are written in C/C++. These process untrusted input — media files, network data. Stagefright (2015) was a famous bug here: a malformed MP4 triggered a buffer overflow in the media library.
Layer 4 — Hardware Abstraction Layer (HAL)
OEM-specific code that bridges hardware to Android. Each manufacturer writes their own HAL. Quality varies — this is why Samsung phones behave differently from Pixel phones at the hardware interface level. HAL bugs can expose hardware-level privileges.
Layer 5 — Linux Kernel ⚡ CRITICAL TARGET
Controls everything: memory allocation, process scheduling, ALL hardware including USB. Kernel code runs with the highest privilege possible. A bug here means the attacker controls everything — there's nothing more privileged above the kernel. This is why kernel exploits are the most severe category.

SELINUX — THE CROSS-CUTTING CONTROL

SELinux applies on top of ALL layers. Even if you become root (UID 0), SELinux can still block specific actions based on policy labels. This is why modern exploits need a separate "SELinux bypass" step — getting root isn't enough on its own.

THREE EXTRA PROTECTIONS

Verified Boot — Cryptographically checks each partition at startup. Modified system files are detected.
FBE — User data encrypted. Keys derived from PIN + hardware secrets in TEE.
TEE — Trusted Execution Environment: a physically separate processor that holds crypto keys. Even root cannot extract them directly.

WHAT TO SAY

"Android uses layered defense — each layer is independently designed to stop different attacks. For an attacker to get root access via USB, they need to bypass every relevant layer. This is what makes it hard and why the research is interesting — you must chain multiple weaknesses together."

SLIDE 05 USB Attack Surface — Why USB is the Target ▼

WHAT THIS SLIDE SHOWS

USB communication flow diagram, 3 attackable points, and 4 USB input types (HID, MTP, Networking, Composite).

WHY USB IS POWERFUL — THE KEY INSIGHT

When you connect a USB device to Android, the kernel's USB subsystem automatically loads the appropriate driver. This happens at kernel level, BEFORE the lockscreen checks anything, BEFORE any permission dialog appears. The kernel must parse the USB descriptor packets to identify the device — and that parsing happens in C code with full kernel privileges.

USB COMMUNICATION FLOW — EXPLAIN EACH STEP

USB Device→ Sends Descriptors→ Android Loads Driver→ Data Exchange→ Kernel Processing

Step 1 — USB Device connects: Physical connection made. Power flows. Nothing has happened yet software-wise.

Step 2 — Sends Descriptors: Every USB device sends "descriptor packets" — small data structures identifying what kind of device it is (keyboard, storage, phone charger). This is mandatory in the USB protocol. The kernel must receive and parse these.

Step 3 — Android Loads Driver: Based on descriptor parsing, the kernel selects and loads the appropriate driver. This is automatic — no user involvement. The driver runs in kernel space with full privileges.

Step 4 — Data Exchange begins: Driver and device communicate using their protocol (MTP for files, HID for keyboard, etc.).

Step 5 — Kernel Processing: Everything is processed inside the kernel. Any bug here has maximum impact.

3 ATTACKABLE POINTS

Device Descriptors: Send invalid, oversized, or malformed descriptors. The driver must parse them. If parsing is buggy — overflow happens before any data transfer even begins.

MTP Protocol Parsing: MTP uses structured packet headers with length fields. If the driver doesn't validate that the declared length matches actual data, a crafted packet overflows the receiving buffer inside the kernel MTP handler.

State Machine Attacks: USB has a defined sequence of states (connect → enumerate → configure → transfer). Sending commands out of sequence can trigger unhandled states — the driver code doesn't know what to do and behaves unexpectedly.

4 USB INPUT TYPES

HID (Human Interface Device): Pretends to be a keyboard or mouse. Can inject keystrokes — but limited on locked device because the lockscreen intercepts input. Useful as a support vector only.

MTP (Media Transfer Protocol): Used for file transfer between phone and PC. Has complex packet parsing at kernel level. Historically the highest-risk protocol for driver bugs.

USB Networking: Device appears as a USB Ethernet adapter. Can inject network packets — useful for network-level attacks if combined with another vector.

Composite Device: Device presents multiple interfaces simultaneously (e.g. keyboard + storage + network at once). Complex to handle, more driver code paths, larger attack surface.

WHAT TO SAY

"USB is our primary target because it's the one external input that reaches the kernel before the security model fully applies. The moment a USB device connects, the kernel starts processing its descriptor — in C code, at kernel privilege. That's the window we're interested in."

SLIDE 06 Vulnerability Classes — 4 Types of Memory Bugs ▼

WHAT THIS SLIDE SHOWS

Four memory vulnerability classes: Buffer Overflow, Use-After-Free, Race Condition, Integer Overflow — each with how it works, defense, and impact.

Buffer Overflow
A driver allocates a fixed buffer — say 64 bytes. When it copies USB input data into that buffer, it doesn't check if the input is larger than 64 bytes. If input is 200 bytes, the extra 136 bytes overflow into adjacent memory — overwriting whatever is stored there, including function pointers.
char buf[64];
// DANGEROUS: no length check
memcpy(buf, usb_data, usb_len);
// if usb_len=200 → overflow!
Impact: Control execution flow. Defense: Stack canaries, bounds checking, ASLR, safe APIs.
Use-After-Free (UAF)
Memory is allocated, used, then freed. But the pointer variable still holds the old address. If an attacker can allocate their own controlled data into that freed memory region before the dangling pointer is used again, the program reads attacker data instead of legitimate data — and may call attacker-controlled function pointers.
free(obj);
// attacker fills freed memory
obj->func_ptr(); // calls attacker code!
Impact: Hijack vtable / function pointer. Defense: Pointer nullification, heap isolation, memory-safe languages.
Race Condition (TOCTOU)
Time-of-Check to Time-of-Use. A security check runs at time T1 and passes. But between T1 and when the action actually happens (T2), an attacker running a parallel thread changes the state. The check result is now invalid but the action proceeds as if it passed.
if (check_permission()) { // T1: PASS
// attacker changes state HERE
do_privileged_action(); // T2: bypass!
Impact: Bypass security validation. Defense: Mutex locks, atomic operations, kernel spinlocks.
Integer Overflow
A size calculation multiplies user-controlled values. If the result exceeds the maximum integer size, it wraps around to a tiny number. A tiny allocation is made, but data sized by the original large value is written into it — causing a buffer overflow as a second step.
// count=65536, size=65536
int total = count * size;
// wraps to 0 → tiny alloc
// write 4GB into 0 bytes!
Impact: Leads to buffer overflow. Defense: Integer overflow checks, compiler flags, safe math libraries.

WHAT TO SAY

"All four vulnerability classes exist because USB drivers are written in C, which has no automatic memory safety. The programmer must manually check every input length, every pointer lifetime, every arithmetic result. One missed check in millions of lines of code is enough."

SLIDE 07 Attack Chain — 5-Step Proof of Concept ▼

WHAT THIS SLIDE SHOWS

The 5-step attack chain with a "what happens" column and a "why this matters" column for each step.

THE MOST IMPORTANT CONCEPT — READ THIS CAREFULLY

The chain has two columns per step. Left = what the attacker does. Right = why that step is significant. When explaining to an interviewer, always say BOTH — the action AND the reason it matters. This shows depth of understanding.

01
USB Connection
A malicious USB device is connected to the locked Android phone.
Why this matters: USB descriptor parsing happens at kernel level BEFORE the lockscreen or any permission check. The kernel must identify the device — no way to skip this. The phone has no choice but to process the connection.
02
Crafted Input Injection
The attacker device sends malformed USB descriptors or MTP packets with invalid length fields.
Why this matters: The kernel USB driver processes this in C code with no automatic bounds checking. A crafted length field that exceeds the buffer size creates a memory corruption opportunity.
03
Vulnerability Triggered
The malformed input hits an unchecked buffer copy or state machine flaw in the driver.
Why this matters: Memory adjacent to the buffer is overwritten — potentially including function pointers or kernel data structures that control execution flow. The attacker now has influence over the kernel's behavior.
04
Kernel Code Execution
The overwritten function pointer is called → CPU jumps to attacker-controlled memory → attacker code runs.
Why this matters: Code is now running in kernel context — the most privileged execution environment. The attacker can now read/write any kernel data structure, including the process credential structure.
05
Privilege Escalation
The kernel credential structure (task_struct → cred → uid) is modified to UID = 0.
Why this matters: The process is now root. BUT — SELinux still applies. Getting UID 0 is not the end. A SELinux policy bypass is a separate, additional step required for unrestricted access. This is a critical distinction.

WHAT TO SAY

"Each step in this chain is necessary — skip one and the next is impossible. That's exactly why Android's defense-in-depth works: you must defeat 4 to 5 separate mitigations, each designed to stop a different attack class. This chain is theoretically viable under specific conditions — a known unpatched vulnerability in the USB driver — which is why the research focuses on identifying those conditions."

EXPLAIN EACH TOOL IN DETAIL

Facedancer: Python library that turns a Raspberry Pi/laptop into ANY USB device. You write Python code describing how your device behaves — its descriptor, responses, data. Facedancer sends actual USB signals via GreatFET hardware. Used to deliver crafted descriptors to the target.

Ghidra: NSA open-source reverse engineering tool. Decompiles kernel driver binaries to pseudo-C code. You look for dangerous patterns: memcpy without length checks, user-controlled size values, unchecked pointer dereferences. It's the tool you use to READ the driver code and FIND the bug before triggering it.

Frida: Injects JavaScript into a running Android process. You hook a function and every time it's called, you see the arguments. Used AFTER finding a bug in Ghidra to CONFIRM the bug is reachable at runtime — "does my crafted USB input actually reach this vulnerable function?"

ADB (Android Debug Bridge): Official Google tool for communicating with Android devices. Used in Phase 1 on an unlocked test device to explore the filesystem, understand structure, compare behavior when locked vs unlocked. NOT part of the attack — purely for research understanding.

ExploitDB / CVE databases: Public database of disclosed vulnerabilities. Search for "Android USB kernel CVE" to find past bugs — learn what components were affected, how bugs were found, what the patch looked like. This guides where to look for new bugs.

dmesg / lsusb: Linux kernel log tools. When you connect a USB device, dmesg shows exactly which driver loaded, what errors occurred, and any crashes. lsusb shows all connected devices. These are your observation window into kernel behavior.

WHAT TO SAY

"These tools form a complete research pipeline. You start with ADB to understand normal behavior, use Ghidra to find the dangerous code statically, Facedancer to deliver the crafted input to the target, and Frida plus dmesg to verify what actually happens at runtime. No single tool does everything — they work as a chain."

SLIDE 10 Android Defense-in-Depth — Why Root Isn't Enough ▼

WHAT THIS SLIDE SHOWS

6 specific mitigations: KASLR, Stack Canaries, SELinux, CFI, FBE+TEE, Verified Boot — each with a detailed explanation of exactly what it does and why it matters.

THE CRITICAL INSIGHT — SAY THIS TO JUDGES

"Getting root (UID 0) is NOT the end of the chain. SELinux still enforces policy. FBE+TEE still encrypts user data. Verified Boot detects persistent changes. Modern Android requires chaining at minimum 4 separate bypasses to achieve unrestricted data access. This is defense-in-depth working as intended."

EACH DEFENSE EXPLAINED

KASLR (Kernel Address Space Layout Randomization): Every boot, the kernel's code is loaded at a random memory address. Even if you have code execution, you don't know WHERE the kernel is in memory. You'd need to first "leak" a kernel address (from another bug) to calculate the base offset before you can redirect execution anywhere useful.

Stack Canaries: A random value is placed on the stack between local variables and the return address. When a function returns, it checks if the canary changed. A buffer overflow that overwrites the return address also overwrites the canary — mismatch is detected, program crashes instead of executing attacker code.

SELinux (Mandatory Access Control): Every process has a security context label (e.g. "untrusted_app"). Every file, socket, and resource also has a label. SELinux only allows interactions explicitly permitted in policy. Even UID 0 (root) cannot perform actions outside its SELinux context. This is why attackers need a separate "SELinux bypass" — escalation to UID 0 is step 1, SELinux bypass is step 2.

CFI (Control Flow Integrity): The compiler pre-computes all valid function call targets. At runtime, any indirect call (through a function pointer) is checked against this list. ROP chains that chain small code gadgets together are blocked because they jump to addresses not in the pre-approved list.

FBE + TEE: File-Based Encryption uses keys derived from PIN + hardware secrets. The hardware secrets never leave the TEE — a physically separate processor. Even with root access, you cannot read user data without the correct PIN, because the decryption keys are inside a separate chip you can't directly access.

Verified Boot (AVB): At every boot, each partition's hash is verified against a cryptographic signature. If a system file was modified (to persist an exploit), the hash mismatch is detected. The device won't boot normally — showing a warning or refusing to start. This prevents "persistent" compromise even after a successful exploit.

WHAT TO SAY

"Each of these mitigations addresses a specific step in the attack chain. KASLR makes it hard to redirect execution. Stack canaries detect the overflow. SELinux restricts even root. CFI blocks ROP chains. FBE+TEE protects data even after root. Verified Boot prevents persistence. The attacker must bypass each one separately — that's why modern Android requires a chain of 4 or 5 separate vulnerabilities, not just one."

SLIDES 11–14 Mitigations, Disclosure, Limitations, Conclusion ▼

SLIDE 11 — MITIGATION RECOMMENDATIONS

Kernel/USB Driver: Validate ALL USB descriptor lengths before processing. Adopt Rust for new kernel modules — it eliminates memory safety bugs by design. Android 13+ already supports Rust kernel modules.

USB Access Policy: Default locked screen to charging-only USB mode. Require explicit unlock before MTP or ADB activates. Many OEMs already do this.

OEM Diagnostics: Disable EDL/Meta Mode without hardware auth token. Remove pre-authorized ADB keys from production builds.

General: SELinux enforcing on all production builds. Monthly kernel security patches minimum.

SLIDE 12 — RESPONSIBLE DISCLOSURE (6 PHASES)

Discover→ Verify→ Document→ Notify→ Wait 90 days→ Publish

The 90-day window is the Google Project Zero standard. You notify the vendor privately, give them time to patch, then publish publicly. This benefits everyone — vendor gets time to fix, public gets information after it's safe.

SLIDE 13 — LIMITATIONS

No universal exploit: Every OEM has a different USB driver implementation. A technique for Samsung's driver won't work on Pixel's.

Requires a specific vulnerability: Without an unpatched CVE, the chain is theoretical. Finding a new bug requires deep kernel fuzzing research.

SELinux as final barrier: Even after UID 0, SELinux may block the specific actions needed for the demo.

WHAT TO SAY FOR CONCLUSION

The key insight is that modern Android's defense-in-depth means no single vulnerability gives full access — you must chain four or five separate weaknesses, which significantly raises the cost of real-world exploitation."

PREPARE THESE ANSWERS

These are the most common questions judges and interviewers ask about this kind of security research project. Practice saying each answer out loud.

What is the main research contribution of your project?
Our contribution is a structured analysis of Android's USB attack surface framed as a complete exploit chain. We mapped all 6 security layers, identified USB as the highest-value entry point because kernel parsing occurs before the security model applies, designed a theoretical 5-step exploit chain, mapped each step to the specific mitigation that defeats it, and produced concrete engineering recommendations per layer. We also identified a gap in existing tooling — current tools work in isolation — and proposed a unified orchestration framework as future work.
Why USB specifically? Why not Bluetooth or WiFi?
USB is uniquely powerful because kernel-level descriptor parsing happens before any user interaction or permission check. When a USB device connects, the kernel must identify it — this is mandatory in the USB protocol and happens automatically regardless of lockscreen state. Bluetooth requires a pairing sequence with user approval. WiFi requires network association. USB has no such gatekeeping at the kernel level, which is what makes it the highest-value theoretical attack surface for this research.
Does getting root (UID 0) give you access to all user data?
No — and this is a critical distinction. Root (UID 0) and unrestricted data access are two different things on modern Android. Even as root, SELinux enforces mandatory access control based on process labels, not just UID. Additionally, user data is encrypted with File-Based Encryption (FBE) using keys derived from the user's PIN combined with hardware secrets in the TEE. Those keys never leave the TEE even under root access. So root gives you kernel control — it does NOT automatically give you readable user files. A full attack chain would need: UID escalation → SELinux bypass → device unlock to obtain FBE keys.
What is SELinux and why does it matter even after root?
SELinux is Mandatory Access Control — it's a system where every process and every resource has a security context label. The kernel enforces a policy that only allows specific label-to-label interactions. Critically, these rules apply regardless of UID. Even a process running as root (UID 0) cannot perform actions that its SELinux label isn't permitted. For example, a process with context "init" cannot access files labeled for "system_server". This is why modern exploits need a separate SELinux bypass step — simply achieving UID 0 isn't enough.
What is the difference between static and dynamic analysis?
Static analysis (Ghidra) — you analyze the binary code without running it. You decompile the driver to pseudo-C and manually review for dangerous patterns: unchecked memcpy, user-controlled sizes, missing length validation. You find WHERE the bug might be. Dynamic analysis (Frida) — you hook into the running program and observe what actually happens when specific inputs arrive. You confirm that your crafted input actually reaches the vulnerable function at runtime. Static analysis finds the bug in theory; dynamic analysis confirms it's reachable in practice.
What is Facedancer and how does it differ from BadUSB?
Facedancer is a Python library for scripting arbitrary USB device behavior. You define exactly how your device responds to every USB request — its descriptors, data, protocol behavior. It uses GreatFET hardware to translate Python instructions into actual USB signals. It's a fully programmable research platform. BadUSB is a specific attack concept where a device pretends to be a keyboard and injects keystrokes — it's one specific use case. Facedancer is the framework that can implement BadUSB behavior and much more — MTP fuzzing, state machine attacks, protocol manipulation. Think of Facedancer as the platform and BadUSB as one of many possible scripts you could run on it.
What would your unified tool do that existing tools don't?
Current tools require manual handoff between stages — you run Wireshark, manually analyze output, configure Facedancer separately, manually inspect dmesg, write Frida scripts independently. A unified framework would automate the pipeline: automatically parse device profiles from USB capture, feed them to Facedancer, instrument Frida hooks based on Ghidra's static analysis output, monitor for crashes, and generate a structured finding report — all from a single command. The innovation is orchestration and automation, not new exploitation techniques. This reduces the expertise threshold for USB security research and enables systematic analysis across many devices.
Why is this research ethical? Isn't this just teaching people to hack?
Security research follows the principle that defenders need to understand attacks to build effective defenses. All research was conducted on our own test devices — no unauthorized access. The attack chain we describe is theoretical and requires a specific unpatched vulnerability to execute. Our deliverables explicitly include mitigation recommendations and a responsible disclosure framework. The same knowledge that explains how an attack works is what enables engineers to fix the underlying flaw. Published security research — including detailed CVE write-ups from Google Project Zero — follows this exact model and has directly improved Android's security over time.
